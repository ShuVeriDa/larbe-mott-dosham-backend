# Описание проекта «Мотт Ларбе Дошам» для дизайна

Документ предназначен для создания дизайна продукта в отдельном проекте (Claude и др.). Содержит точное описание текущего состояния backend, целевой функциональности и карты страниц.

---

## 1. Суть продукта

**Название:** Мотт Ларбе Дошам (объединённый словарь чеченского языка).

**Цель:** Единый высококачественный цифровой словарь чеченского языка, объединяющий 14 словарных источников (12 существующих словарей + ручной сборник + неологизмы), с морфологическим анализом, склонением, спряжением и API для разработчиков.

**Ключевая идея:** Один словарь вместо четырнадцати. Каждое слово обогащено грамматикой, примерами, фразеологией и литературными цитатами из всех источников. Поиск работает в обоих направлениях (чеченский → русский, русский → чеченский) с учётом морфологии.

**Решаемые проблемы:**

- Отсутствие единого цифрового словаря чеченского языка
- Разрозненность данных по 14 разным словарям с разным форматом и покрытием
- Сложность чеченской морфологии (4 именных класса, 8 падежей, эргативный строй)
- Отсутствие инструментов склонения и спряжения для чеченского языка
- Отсутствие API для разработчиков языковых приложений

## 4. Модель данных (основные сущности)

### Словарная запись (UnifiedEntry)

- **id:** autoincrement
- **word:** чеченское слово (основная форма)
- **wordNormalized:** нормализованная форма для поиска
- **wordAccented:** с ударениями/тильдами
- **partOfSpeech:** часть речи на русском (сущ., гл., прил., ...)
- **partOfSpeechNah:** часть речи на чеченском (ц1ердош, хандош, ...)
- **nounClass:** грамматический класс ед.ч. (ву, йу, ду, бу)
- **nounClassPlural:** грамматический класс мн.ч. (если отличается)
- **grammar:** JSON — падежные формы, глагольные формы (genitive, dative, ergative, instrumental, plural, verbPresent, verbPast, ...)
- **meanings:** JSON — `[{ translation, examples: [{nah, ru}] }]`
- **phraseology:** JSON — `[{ nah, ru }]` фразеологизмы
- **citations:** JSON — `[{ text, source? }]` литературные цитаты
- **latinName:** латинское анатомическое название
- **styleLabel:** стилистическая помета (Устаревшее, Просторечное, Поэтическое, ...)
- **variants:** `String[]` — варианты написания
- **domain:** предметная область (anatomy, computer, math, law, geology, sport)
- **cefrLevel:** уровень сложности (A1–C2), присвоенный эвристикой
- **entryType:** standard | neologism
- **sources:** `String[]` — из каких словарей (maciev, baisultanov, ...)
- **createdAt, updatedAt**

### Пользователи и роли

- **User:** id (uuid), email, password (Argon2), username, name, hashedRefreshToken, status (active | blocked)
- **UserSession:** id, userId, ipAddress, userAgent, createdAt, revokedAt
- **Role:** name — `USER`, `EDITOR`, `ADMIN`
- **Permission:** code — `CAN_EDIT_ENTRIES`, `CAN_ADD_ENTRIES`, `CAN_DELETE_ENTRIES`, `CAN_MANAGE_USERS`, `CAN_MANAGE_API_KEYS`, `CAN_RUN_PIPELINE`
- **UserRoleAssignment:** userId + roleId
- **RolePermission:** roleId + permissionId

### API-ключи

- **ApiKey:** id, key, name, role (USER | EDITOR | ADMIN), isActive, lastUsedAt

### Пользовательские функции

- **UserFavorite:** userId + entryId (уникальная пара)
- **SearchHistory:** id, userId, query, lang (nah | ru | unknown), createdAt
- **Suggestion:** id, userId, entryId, field, oldValue, newValue, comment, status (PENDING | APPROVED | REJECTED), reviewedBy, reviewedAt, reviewComment

### Аудит

- **EntryEditLog:** id, entryId, userId?, apiKeyId?, action (update | bulk_update), changes (JSON: `{ field: { old, new } }`), createdAt

## 6. API (краткая сводка для дизайна)

Базовый префикс: **`/api`**. Защищённые маршруты: **`Authorization: Bearer <access_token>`** или **`X-API-Key: <key>`**. Refresh — в httpOnly cookie.

### Авторизация и профиль

| Метод | Путь                           | Auth   | Назначение                   |
| ----- | ------------------------------ | ------ | ---------------------------- |
| POST  | `/api/auth/register`           | —      | Регистрация (5 req/min)      |
| POST  | `/api/auth/login`              | —      | Вход (5 req/min)             |
| POST  | `/api/auth/login/access-token` | Cookie | Обновление access по refresh |
| POST  | `/api/auth/logout`             | Bearer | Выход                        |

**Регистрация (body):** `{ email, password, username, name }`. Пароль: мин. 6 символов, 1 заглавная, 1 спецсимвол. Username: 2–16 символов.

**Ответ login/register:** `{ user: { id, email, username, name }, accessToken }` + refresh в cookie (httpOnly, 7 дней).

| ----- | -------------------------------- | ---- | ------------------------------------------------------- |

**Логика поиска:**
- Определение языка: наличие Ӏ или тильды → чеченский, кириллица → русский
- Чеченский запрос: поиск по `wordNormalized` с `ILIKE` + ранжирование `similarity()`
- Русский запрос: поиск в JSON `meanings::text` с `ILIKE`
- Если чеченский поиск не дал результатов — попытка лемматизации (убрать падежные окончания и найти базовую форму); в `meta.lemmaHint` возвращается подсказка
- Кэширование в Redis (5 мин TTL)

## 7. Ключевые сценарии для UI/UX

### 7.1 Поиск слова (основной сценарий)

1. Пользователь вводит слово в поисковую строку (чеченское или русское).
2. Frontend вызывает `GET /api/dictionary/search?q=...` с опциональными фильтрами.
3. Результаты отображаются списком: слово, часть речи, класс, первое значение, CEFR-уровень, источники.
4. Клик по записи → карточка слова (`GET /api/dictionary/:id`).
5. Карточка содержит: все значения с примерами, фразеологию, цитаты, грамматику, варианты написания, стилистическую помету.
6. Если пользователь авторизован:
   - Кнопка «В избранное» → `POST /api/favorites/:entryId` (toggle)
   - Запрос автоматически сохраняется в историю поиска
   - Кнопка «Предложить правку» → форма создания Suggestion

### 7.2 Склонение и спряжение

1. На карточке слова: если `partOfSpeech` = существительное → секция «Склонение» (`GET /api/dictionary/declension/:word`).
2. Таблица 8 падежей × 2 числа (единственное / множественное).
3. Если `partOfSpeech` = глагол → секция «Спряжение» (`GET /api/dictionary/conjugation/:word`).
4. Таблица 9 временных форм + причастия + деепричастия + повелительное наклонение + отрицание.

### 7.3 Случайное слово / Слово дня

1. На главной странице или отдельной секции: «Слово дня» (`GET /api/dictionary/random?cefr=A1`).
2. Отображается карточка со словом, значением и примером.
3. Кнопка «Ещё слово» → повторный запрос.

### 7.4 Фразеология

1. Пользователь вводит запрос в разделе «Фразеология» (`GET /api/dictionary/phraseology?q=...`).
2. Результаты: записи, содержащие совпадающие фразеологизмы.
3. Каждая запись показывает найденные фразеологизмы с выделением совпадений.

### 7.5 Избранное

1. Список избранных записей: слово, перевод, дата добавления.
2. Клик → карточка слова.
3. Кнопка удаления из избранного (toggle).

### 7.6 История поиска

1. Список недавних запросов: текст запроса, язык (чеченский/русский), время.
2. Клик по запросу → повторный поиск.
3. Кнопка «Очистить историю».

### 7.7 Предложения правок

1. Пользователь на карточке слова видит кнопку «Предложить правку».
2. Форма: выбор поля (значения, слово, класс и т.д.), новое значение, комментарий.
3. Отправка → `POST /api/suggestions`.
4. В разделе «Мои предложения» — список со статусами (PENDING / APPROVED / REJECTED).

### 7.8 Admin: управление пайплайном

1. Страница статуса: `GET /api/merge/status` — сколько словарей распарсено, сколько записей в unified, сколько в БД.
2. Лог слияния: `GET /api/merge/unified-log` — шаги, оставшиеся словари, рекомендация следующего шага.
3. Действия: кнопки «Parse», «Unify Step», «Load», «Improve», «Rollback», «Reset» с подтверждением.
4. Результат каждого действия отображается inline: количество новых/обогащённых слов, пропущенных и т.д.

### 7.9 Admin: качество данных

1. Dashboard со статистикой: процент записей без значений, без класса, без POS, без примеров.
2. Таблица проблемных записей с фильтром по типу проблемы.
3. Клик по записи → переход к редактированию.
4. Редактирование записи: форма со всеми полями, JSON-редактор для meanings/grammar.

### 7.10 Admin: модерация предложений

1. Список предложений с фильтром по статусу.
2. Каждое предложение: автор, запись, поле, старое и новое значение, комментарий.
3. Кнопки «Approve» / «Reject» с возможностью добавить комментарий модератора.

### 7.11 Admin: пользователи

1. Таблица пользователей: email, username, роли, статус.
2. Карточка пользователя: профиль, назначенные роли, действия (блокировка, назначение роли).

### 7.12 Admin: API-ключи

1. Таблица ключей: имя, роль, статус (активен/неактивен), дата последнего использования.
2. Создание: имя + роль → генерация ключа.
3. Редактирование: смена имени, роли, деактивация.

## 8. Уже реализовано на backend

- Регистрация и вход (JWT access 1ч + refresh 7д в httpOnly cookie), Argon2, сессии.
- RBAC с 3 ролями (USER / EDITOR / ADMIN) и 6 разрешениями.
- API-ключи с ролями для межсервисного доступа (`X-API-Key`).
- Rate limiting: 100 req/min глобально, 5 req/min на auth.
- **Поиск** (двуязычный): чеченский → `wordNormalized` с trigram similarity, русский → поиск в `meanings` JSON. Фильтры: cefr, pos, nounClass, entryType. Пагинация. Кэш в Redis (5 мин).
- **Лемматизация** (fallback): если поиск не дал результатов — попытка убрать падежные окончания и найти базовую форму.
- **Склонение**: 4 типа, 8 падежей, единственное и множественное число. Автоопределение типа по инструменталису.
- **Спряжение**: 2 типа, 9 времён, причастия, деепричастия, повелительное наклонение, отрицание. Генерация из 2 базовых форм (present + perfect).
- **Случайное слово** (`/random?cefr=A1`).
- **Поиск по фразеологизмам** (`/phraseology?q=...`).
- **Статистика словаря** (`/stats`): total, domains, cefrLevels.
- **CRUD записей**: обновление одной записи и bulk-обновление (до 100). Все изменения логируются.
- **Избранное**: toggle, список, проверка.
- **История поиска**: автозапись, просмотр, очистка.
- **Предложения правок**: создание, просмотр своих, модерация (approve/reject).
- **ETL-пайплайн**: 14 парсеров, слияние с дедупликацией по ключу `word + homonymIndex + nounClass`, снапшоты (возможность rollback), загрузка в БД с GIN-индексом, очистка данных (improve).
- **Admin: пайплайн** через API (parse, unify-step, load, improve, rollback, reset).
- **Admin: качество данных** — статистика и поиск проблемных записей.
- **Admin: пользователи** — список, назначение ролей, блокировка/разблокировка.
- **Admin: API-ключи** — CRUD.
- **Admin: аудит-лог** — история изменений записей.
- **Мониторинг пайплайна**: статус, превью, лог шагов.
- **CEFR-уровни**: эвристическая оценка при загрузке (A1–C2).
- **Swagger** документация в dev-режиме (`/api/docs`).
